  WHAT IS THIS?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The REPUBLICKA Live Chamber is a real-time chat room built as a single PHP
file. It was designed and built in approximately 20 minutes in a single
conversation between Wendell Charles NeSmith — philosopher, author, musician,
and founder of REPUBLICKA — and an AI assistant.

It lives at kkk.education/chat/ and is released under CC0 Public Domain.
Anyone can use it, copy it, sell it, or build on it with zero permission
required.

The tagline writes itself: one person and one AI session produced a working
replacement for Discord, IRC, Slack, and every other chat platform — in less
time than it takes to create a Discord account.

  THE PHILOSOPHY BEHIND IT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Most chat platforms exist to extract value from their users. Accounts exist
to track you. Complexity exists to create dependency. Scale exists to justify
valuation. Every friction point — phone number verification, age gates,
invite systems, moderation queues — serves the platform, not the community.

The Live Chamber asks a different question:

     What does a human actually need to talk to another human?

The answer is three things:

     - A name
     - A shared URL
     - A text box

Everything else is overhead built to serve extraction, not conversation.

This is the REPUBLICKA principle applied directly: platform sovereignty, no
gatekeepers, no surveillance, no landlord who can evict your community at
will.

  HOW IT WORKS — THE ARCHITECTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ◈ SERVER: RELAY ONLY

The server holds messages for only 5 minutes — just long enough for active
users to receive them via polling. After 5 minutes they are gone from the
server permanently. There is no permanent server-side message store. The
server is a relay, not a database.

  ◈ CLIENT: LOCALSTORAGE IS THE REAL STORE

Your browser's localStorage is where messages actually live permanently.
Every message you receive while online is saved to your device automatically.
There is no cap on how many messages you can store. You clear your history
when you want, using a built-in Clear button that asks for confirmation first.

The design philosophy: if you weren't in the room when a message was sent and
it has expired from the relay, you don't see it. This is intentional. The
chat is present-tense and community-oriented, not an archive.

  ◈ POLLING, NOT WEBSOCKETS

Rather than requiring Node.js, Redis, or persistent WebSocket connections,
the chat uses simple HTTP polling every 2.5 seconds. The client asks the
server: "anything new since my last check?" This works on any PHP host with
zero additional infrastructure. It is the simplest possible approach that
actually functions.

  FEATURES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ◈ JOIN WITHOUT AN ACCOUNT

Users type a name and enter. No email. No password. No phone number. No
CAPTCHA. No waiting for moderator approval. No terms of service. The name is
saved to localStorage so returning users don't have to retype it.

  ◈ USER COLOURS

Each username is hashed to one of 12 distinct colours:

     blue · green · red · purple · amber · teal · orange
     periwinkle · lime · pink · mint · yellow

The same name always gets the same colour on every device for everyone in the
room. The logged-in user always appears in gold. Colours apply to the avatar,
the username, and the avatar border simultaneously — making conversations
readable at a glance.

  ◈ AUTOMATIC HYPERLINKS

Any URL typed in a message automatically becomes a clickable link that opens
in a new tab. The link colour matches the sender's user colour for visual
consistency.

  ◈ LIGHT AND DARK MODE

A single button in the header toggles between:

     Dark mode  — black background, gold text (the REPUBLICKA aesthetic)
     Light mode — warm cream with deep gold

Preference saved to localStorage and persists across sessions.

  ◈ ONLINE USERS SIDEBAR

A sidebar shows who is currently online, with each user's name and dot
displayed in their assigned colour. Presence is tracked via heartbeat — users
drop off the list approximately 20 seconds after they stop being active.

     Desktop — sidebar always visible on the right
     Mobile  — slides in from the right when the users button is tapped

  ◈ STORED HISTORY WITH CLEAR BUTTON

When you join, your past messages load instantly from localStorage under a
'Stored History' divider, labelled so you can distinguish local history from
live messages. A Clear button — protected by a confirmation dialog — wipes
your local history entirely when you want a clean slate.

  ◈ DEDUPLICATION

A Set tracks every rendered message by timestamp and username. If the same
message arrives in multiple polls it is silently skipped. This prevents the
double-post issue that can occur when a send confirmation poll and an interval
poll overlap.

  ◈ SERVER CLOCK SYNC

On joining, the client makes a sync call to get the server's current
timestamp and uses that as the poll cursor — rather than the client's local
clock. This eliminates a clock-skew bug where messages sent immediately after
joining would be silently dropped because the client clock was behind the
server clock.

  ◈ RATE LIMITING

IP-based rate limiting prevents more than one message per 1.5 seconds from a
single user. This stops spam without requiring accounts or any user-visible
mechanism.

  ◈ ADMIN PANEL

A password-protected admin panel accessible via a button in the header
provides four tabs:

     Stats    — relay count, online count, ban count, clear-relay button
     Messages — all live relay messages with individual delete buttons
     Users    — who is online with kick and ban buttons
     Bans     — add ban by username, list all bans with unban buttons

The admin password is stored in memory only and clears when the tab is
closed.

  ◈ MOBILE FIRST

     - 100dvh for correct height on iOS Safari
     - safe-area-inset-bottom padding for home bar clearance
     - 44px minimum tap targets throughout
     - Sliding sidebar panel on small screens

  ◈ ENTER / LEAVE ANNOUNCEMENTS

System messages are automatically posted when a user joins or leaves. These
appear as small labelled pills in the message stream rather than full message
bubbles, keeping the visual hierarchy clean.

  TECHNICAL STACK
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The entire system is a single file. Complete technical stack:

     - PHP 7.4+              backend API and HTML page server
     - chat-relay.json       messages (5 min TTL)
     - chat-online.json      presence heartbeat
     - chat-bans.json        ban list
     - Vanilla JavaScript    no frameworks, no dependencies
     - localStorage          permanent client-side message store
     - HTTP polling          every 2.5 seconds, no WebSockets
     - Cinzel + Crimson Text Google Fonts via CDN
     - CSS custom properties full light/dark theme system

     No database.
     No Node.js.
     No Redis.
     No build step.
     No npm.
     No webpack.

  WHAT MAKES IT DIFFERENT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ◈ VS DISCORD

Discord requires an account, phone number verification on many servers, terms
of service agreement, and frequently bans users or entire communities with
limited appeal. Many servers require bot-based verification, invite links, and
moderator approval. Discord can deplatform any community at any time for any
reason.

The Live Chamber requires nothing except a name. The host owns the platform
completely and cannot be deplatformed.

  ◈ VS IRC

IRC pioneered the no-account chat concept in 1988 and is still running. The
Live Chamber is spiritually descended from IRC. The key difference: IRC
requires a dedicated client application. The Live Chamber is a URL. Any
browser, any device, no installation.

  ◈ VS SLACK

Slack requires an account, an email address, a workspace invitation, and a
subscription for most useful features. Message history is paywalled above a
certain count. The Live Chamber has no paywalls, no message history limits on
the client side, and no subscription.

  THE BUILD STORY — 20 MINUTES TO PRODUCTION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The Live Chamber was built iteratively in a single AI-assisted session:

     1. Decided on flat-file PHP polling — simplest possible option,
        no server configuration required

     2. Built v1: working single-file chat with join screen, message
        polling, system announcements, and IP rate limiting

     3. Added localStorage history, light/dark mode, online sidebar,
        logout, clear history, and mobile layout in v2

     4. Fixed clock-skew bug causing messages to be silently dropped
        for the first minute after joining

     5. Fixed double-post bug using a deduplication Set

     6. Added 12-colour user colour system with hash-based assignment

     7. Added automatic hyperlink detection and rendering

     8. Added admin panel with stats, message management, user
        management, and ban system

Each iteration took minutes. The core insight driving the speed: most of the
complexity in existing chat platforms exists to serve the platform's business
model, not the user's communication need. Strip that away and the technical
problem is trivially simple.
